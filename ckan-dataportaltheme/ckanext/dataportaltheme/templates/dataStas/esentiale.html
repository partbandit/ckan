    Ghidul calității datelor pentru datele tabulare

    All datasets must meet these data quality requirements before publishing them on dataportal.ro.

    • Essential principles
    • Structure for tabular data
    • Other resources

    1. Electronic legibility

    Computer-readable information is given in a format that can be understood by a computer. This means that all relevant fields can be extracted and analyzed without a human contribution. Just because a data set is digitally accessible does not necessarily mean that it is readable for the computer (more information here). In the spirit of open data, data sets must be provided in unprotected file formats. Tabular data is provided as CSV or JSON format. Metadata should be excluded from the data files. These will be included in a zip file with the data file (see Data Pack below).

    2. Ordered data

    The concept of ordered data is based on the suggestion of Hadley Wickham, author of the popular packets Rggotot2 and plyr.

    Hadley proposes three principles:

    • Each column is a variable. This means that each column must have the same unit of measure
    • Each row is an observation. Note that this does not mean that there is only one variable observed for each row. There may be several variables observed for each row: "An observation contains all the values ​​measured on the same unit (such as a person, a day or a race) for all attributes." (Wickham, p. 3)
    • One type of observation unit for each table. For example, observations on the fertility indices of the entire female population and that women of different age groups should be in two tables:
      • Total Fertility Index ", where the observation unit is the entire female population
      • Total fertility index by age group ", where the observation unit is a female population of different age groups

    3. Consistency

    Each variable must be processed and documented in the same way for all datasets. Examples include:

    • Data formats
    • Null and negligible values
    • Unit of measurement

    This makes it easy for users to compile and use more than one set of data, even from different agencies.

    4. Granularity and accuracy

    As far as possible, agencies should provide raw, granular data instead of aggregate and process data, such as percentages. Totals and subtotals should be in separate tables if necessary. For example, there are cases where aggregate numbers (e.g., totals, indices) can not be derived from granular data points.

    5. Lizibilitatea umană

    În final, seturile de date ar trebui prezentate, de asemenea, într-un mod în care are sens pentru utilizatorii umani. Din acest motiv, am creat un set nou de specificații privind metadatele împreună cu acest Ghid de calitate a datelor, pentru a ne asigura că seturile de date sunt documentate în mod clar. De pildă, fiecare tabel trebuie să fie însoțit de o schemă, care precizează tipul de variabilă. Ar trebui să se utilizeze un limbaj simplu pentru a denumi seturile de date și elementele din ele în măsura în care este posibil. Numele de coloane ar trebui să fie descriptive și semnificative. Tot jargonul trebuie explicat în mod clar și, după caz, trebuie furnizate linkuri de referință.

    Pachete de date

    We have adapted this concept from the Open Knowledge International Data Pack specification, which allows us to add metadata and data files to a package. We chose to provide metadata in the YAML format instead of the JSON format because YAML can be read both man and computer. When users download a data set from dataportal.ro, the zip file contains:

    • A text file (in YAML syntax)
    • The resource file (s)
